# KONTi Dashboard MVP

## Overview

The KONTi Dashboard is a bilingual (EN/ES) project management and client dashboard for KONTi Design | Build Studio, a Puerto Rico-based sustainable architecture firm specializing in shipping container construction. This project provides a comprehensive tool for managing projects, client interactions, and internal operations, offering a streamlined approach to project oversight and client engagement for sustainable architecture projects.

## System Architecture

The project is structured as a pnpm workspace monorepo utilizing TypeScript.

### UI/UX Decisions

- **Bilingual Support:** The dashboard supports both English and Spanish, with all user-facing strings on the calculator surface wrapped in an i18n helper for dynamic language switching.
- **Variance Report:** The Calculator's Variance tab displays estimated, invoiced, and actual costs per bucket and material category, with delta pills for `Δ vs Invoiced` and `Δ vs Estimated`. Unassigned invoices are categorized, and the report includes total estimated, invoiced, and actual figures. Client views hide all invoiced data.
- **Role-Aware Project Card Images:** Project cards display `liveCoverImage` for KONTi staff and curated brand mockup images with progress milestones for clients.
- **Branding:** Uses a specific color palette (`#1C1814`, `#E6EAEB`, `#778894`, `#4F5E2A`), Montserrat and Cormorant fonts, and `@assets/Horizontal02_WhitePNG_*` or `@assets/Horizontal02_VerdePNG_*` logos.
- **Responsive Layout:** Designed for phone, tablet, and desktop, with responsive adjustments for offsets, gutters, card grids, list rows, tables, hero sections, chat panels, and page headings.

### Technical Implementations

- **Monorepo Tools:** pnpm workspaces, Node.js v24, TypeScript v5.9.
- **API:** Express 5 framework, with Orval for API codegen from OpenAPI spec, and esbuild for CJS bundling. Runs on port 8080.
- **Authentication:** Token-based authentication using `localStorage` for credentials (`konti_auth`) and language preference (`konti_lang`).
- **Project Report:** Editable report header date persisted in `localStorage`, with server-side PDF generation.
- **Security:** `osv-scanner` for dependency auditing.
- **API Server (`artifacts/api-server`):** Provides routes for authentication, projects, tasks, weather, documents, calculations, receipts, materials, dashboard summary, activity, and AI chat. Includes receipt OCR processing via PDF.co and integrates with Anthropic API for Claude AI chat. Uses static seed data for 3 projects.
- **Lifecycle Persistence (projects, tasks, leads, inspections, change orders, user profiles, notifications-seen, structured vars, assisted budgets, csv mappings, pre-design checklists, project activities):** All project-lifecycle state persists to Postgres via Drizzle (`lib/db/src/schema/lifecycle.ts`), mirroring the estimating playbook. The storage adapter (`artifacts/api-server/src/lib/lifecycle-store.ts`) exposes per-store save helpers and a whole-snapshot `loadLifecycleSnapshotFromDb` / `saveLifecycleSnapshotToDb`. The persistence layer (`artifacts/api-server/src/lib/lifecycle-persistence.ts`) keeps a per-store-key serialised queue and a `persist*` API plus the `appendActivityAndPersist` helper used by every activity-emitting handler. Mutating routes in `routes/projects.ts`, `routes/leads.ts`, `routes/auth.ts`, and `routes/notifications.ts` `await` the matching `persist*` before sending their 200/201 response (500 with `persist_failed` on commit error). On boot, `bootstrap()` awaits `ensureLifecycleHydrated()` alongside the estimating + calculator hydrators; a one-time seed migration (`migrateLifecycleSeedIfNeeded`, idempotency keyed by `lifecycle-seed-2026-05` in `lifecycle_migrations`) imports the in-memory `seed.ts` content into Postgres on first boot and is protected by a clobber guard that refuses to overwrite a non-empty DB whose marker was lost. The in-memory `seed.ts` is NEVER edited at runtime — it remains the canonical bootstrap fixture, while all live edits live in Postgres. **Task #150 — project documents persistence:** `project_documents` (lib/db/src/schema/lifecycle.ts) backs the `DOCUMENTS` map with typed columns for the fields routes read directly + a `metadata` jsonb for the long-tail (`versions[]`, `previewable`, future fields). `POST /projects/:id/documents`, `PATCH /documents/:id`, and `DELETE /documents/:id` now `await persistDocumentsForProject` before responding, so doc uploads, visibility flips, cover-photo toggles, and deletions survive a restart whether or not the Drive integration is configured. Drive (Task #128) remains the canonical store for the file BYTES when configured; this row is the canonical store for the metadata regardless. **Task #147 — accept-lead idempotency across restarts:** the projects table carries a durable `lead_id` column (indexed) stamped by `POST /leads/:id/accept`. The handler no longer relies solely on the in-process `ACCEPTED_LEAD_PROJECTS` map; on cache miss it scans `PROJECTS` for the persisted `leadId` so a re-accept after restart returns the original synthesized project instead of creating a duplicate. If the lead is marked accepted but its project row is gone (operator deletion / orphan), the handler refuses with `409 already_accepted_orphan` rather than silently synthesizing a duplicate.
- **Estimating + Calculator Persistence:** Imported materials, labor rates, project receipts, report templates, contractor estimates, and per-line calculator entries persist to Postgres via Drizzle (`@workspace/db` schema in `lib/db/src/schema/estimating.ts`). Writes are **request-coupled**: every mutating handler `await`s `persistEstimatingState()` / `persistCalculatorEntriesForProject()` before sending its 200 response, so a 200 OK guarantees the row is durably committed (a crash between ack and queue drain cannot lose acknowledged writes). The per-key serialised queue still prevents concurrent writes from racing; `flushEstimatingPersistence()` / `flushCalculatorPersistence()` remain exported for tests that want to drain the queue without issuing another mutation. On commit failure the handler responds 500 with a `persist_failed` payload instead of silently dropping the change. On boot, `bootstrap()` in `artifacts/api-server/src/index.ts` awaits `ensureEstimatingHydrated()` + `ensureCalculatorHydrated()` before `app.listen`, and a one-time JSON migration (`migrateEstimatingJsonIfNeeded`, idempotency keyed by `estimating-json-2026-05` in `estimating_migrations`) imports any legacy `.data/estimating.json` and renames the file to `.migrated.<ISO>` so subsequent boots are no-ops. **Hydration failure policy**: in `NODE_ENV=production` the bootstrap fails fast with `process.exit(1)` so the platform restarts rather than serve seed defaults (the exact silent-data-loss footgun this task eliminates); in dev/test the server logs the error and continues so developers and the test suite aren't blocked when Postgres is intentionally absent. The same policy applies to legacy-JSON migration failures inside `ensureEstimatingHydrated()`: production rethrows to surface "we shipped but skipped unmigrated legacy data" as a hard restart, while dev/test logs and continues so a malformed local file is never a startup blocker. Projects with no calculator rows in Postgres continue to use `seed.ts` defaults. The legacy `lib/estimating-persistence.ts` is kept for one release as a thin deprecated re-export shim — it now delegates `loadEstimatingFromDisk` / `saveEstimatingToDisk` to the new DB-backed `loadEstimatingSnapshotFromDb` / `saveEstimatingSnapshotToDb`, logs a one-time deprecation warning, and converts the previously-sync return values to Promises (Postgres I/O can't be sync). All in-repo callers were migrated to import directly from `./estimating-store` and the shim should be deleted in the release after #141 ships.
- **KONTi Dashboard (`artifacts/konti-dashboard`):** React + Vite SPA with routes for login, dashboard, projects, calculator, materials, and AI. Includes report theme toggles (light, white, dark) persisted per project, and a document upload modal with optimistic removal.
- **Project Lifecycle Model — Macro-Serial / Milestone-Parallel (Task #160 / A-10 decision):** The top-level `PHASE_ORDER` chain (`lead → consultation → pre_design → schematic_design → design_development → construction_documents → permits → construction → completed`, defined in `artifacts/api-server/src/data/seed.ts`) is intentionally **serial** — each `/advance-phase` step in `artifacts/api-server/src/routes/projects.ts` is gated by signatures, payments, OGPE authorization, and an open-punchlist check, so phases cannot legally overlap. **Parallelism lives one level down**: inside the construction macro-phase, the six milestones (foundation, framing, roofing, MEP, finishes, final) carry independent `startDate`/`endDate`/`status` and render as a Gantt-style overlapping timeline (`artifacts/konti-dashboard/src/components/milestones-timeline.tsx`). Future agents: do not relax the macro-serial chain — the gates are the chain. Add new parallel work-streams as additional milestones inside the relevant macro-phase instead.
